# SDD Phase: VALIDATE

> Universal prompt — language-agnostic, works with any AI tool and any tech stack.

## Role

Help the user validate and approve spec blocks that were generated by AI and never formally approved.

## Required Context

Read or have access to:
- `AGENTS.md` — project rules
- `sdd/ANTI_PATTERNS.md` — anti-patterns to avoid

## Arguments

Format: `<project> <spec> [block]`

- **project**: see AGENTS.md for available projects
- **spec**: filename without extension (e.g., `02_DataModel`)
- **block** (optional): block name to validate. If omitted, list available blocks.

## Context

Existing specs in `projectos/<project>/specs/` were generated by AI from initial requirements. They are DRAFT baseline — a starting point, not absolute truth. The user needs to review, adjust and approve each block before implementing.

## Locate Files

1. Spec: `projectos/<project>/specs/<spec>.md`
2. Validation index: `projectos/<project>/VALIDACAO.md`

## Behavior

### Mode 1: Without spec (project only)
Show the project's `VALIDACAO.md` index. Summarize how many blocks are DRAFT vs APPROVED.

### Mode 2: Spec without block
1. Read the indicated spec
2. List all blocks (## sections) with validation status of each
3. Suggest which block to validate next

### Mode 3: Spec with block
1. Read spec and locate the block
2. Read current state in `VALIDACAO.md`
3. If already APPROVED: inform and ask if re-validate
4. Present block content in condensed form:
   - Summary (2-3 sentences)
   - Key points / decisions
   - Edge cases
   - Dependencies
5. **ASK the user**: "Is this block correct as is?"
6. **ADJUST** spec according to feedback
7. **UPDATE** `VALIDACAO.md`: DRAFT -> APPROVED with date

## Verification Checklist

- [ ] Does the block make sense in the project context?
- [ ] Are edge cases realistic?
- [ ] Are dependencies correct?
- [ ] Is there premature generalization? (AP-01, AP-03)
- [ ] Is the block coherent with already APPROVED blocks?
- [ ] Is terminology correct for the domain?

## Rules

- If user changes something that affects other blocks: WARN
- If user removes something: actually delete (don't comment out)
- Never proceed to PLAN with DRAFT blocks

## VALIDACAO.md Format

```markdown
# [Project] - Spec Validation Index

> AI-generated specs as baseline. Each block must be validated by user.

**Last updated**: [DATE]

## Summary

| State | Blocks |
|-------|--------|
| DRAFT | XX |
| APPROVED | XX |
| REMOVED | XX |

## [Spec: 0X_SpecName]

| Block | State | Date | Notes |
|-------|-------|------|-------|
| [block name] | DRAFT | - | - |
| [block name] | APPROVED | YYYY-MM-DD | [notes] |
```

## Integration with SDD

```
New spec:      REQUEST -> SPECIFY -> PLAN -> IMPLEMENT -> CHECK
Existing spec: VALIDATE -> PLAN -> IMPLEMENT -> CHECK
```

## Output

Updated spec block + updated VALIDACAO.md. Inform which blocks remain DRAFT.
